## Bug A — VIS Paid-In transactions wrongly flipped to Paid-Out

### Symptom
These transactions appear in the PDF under the **£ Paid in** column but the
parser was assigning them to `moneyOut` (appearing as negative OTHER expenses):

| Date       | Description                    | Wrong value | Correct      |
|------------|--------------------------------|------------:|-------------:|
| 27/03/2026 | Vis Leo Vegas Newcastle Upo    | -25.00      | INCOME 25.00 |
| 31/03/2026 | Vis Superdrug Stores P Clapham | -10.99      | INCOME 10.99 |
| 08/04/2026 | Vis White Valetta              | -60.00      | INCOME 60.00 |
| 17/04/2026 | Vis White Valetta              | -30.00      | INCOME 30.00 |
| 23/04/2026 | Vis White Valetta              | -100.00     | INCOME 100.00|

**Root cause**: `forceDirectionByType` in `hsbc.ts` was flipping all VIS
transactions from `moneyIn → moneyOut` unless the description contained a
refund keyword.  The Make.com JS parser (module 1248) explicitly does NOT
auto-flip VIS: *"Няма auto-flip за VIS/VISA. Оставяме колоните от PDF/OCR и
balance solver-а да решават."*

**Fix applied** (`apps/api/src/services/processing/parsers/hsbc.ts`):  
Removed the VIS flip from `forceDirectionByType`. Column position from Azure DI
(`c3` = Paid Out, `c4` = Paid In) is now trusted; the balance solver validates
direction via dynamic-programming segment analysis.

## Bug B — Two separate transactions merged into one description

### Symptom
Azure DI sometimes assigns consecutive transactions the same logical row index
(or the second transaction emits no `col=0` date cell), so the parser appended
the second merchant's name and amount to the first transaction:

| Current (merged) description                                    | Should be two rows                                                |
|-----------------------------------------------------------------|-------------------------------------------------------------------|
| Vis 21.Co.Uk 21.Co.Uk Tesco Stores 4342 Streatham              | 21.Co.Uk + **Tesco Stores 4342 Streatham -£13.59**               |
| Vis Oyster Mobile App 0343 2221234 Superdrug Stores P Clapham  | Oyster Mobile App + **Superdrug Stores P Clapham +£10.99**       |
| Welcome Home London Sherwood Food And Mitcham                   | Welcome Home London + **Sherwood Food And Mitcham -£0.50**       |
| Vis White . Tariq Halal Croydo Croydon                         | White + **Tariq Halal Croydon -£2.84**                           |
| Vis Hollister White C London Tesco Stores 5186 London           | Hollister White + **Tesco Stores 5186 London -£1.13**            |

**Root cause**: Continuation rows (no transaction code in col1) that have
description text AND an amount in the Paid-Out/Paid-In column were silently
merged into the current transaction.  The amount was dropped (since
`applyAmountsToTxnFromRow` skips columns already set).

**Fix applied** (`apps/api/src/services/processing/parsers/hsbc.ts`):  
Added an **implicit transaction split** check in the main loop.  When:
- the current transaction already has a movement amount (`txnHasRealAmount`), AND
- the continuation row has non-empty description text (not an amount, not a skip phrase), AND
- the continuation row has an amount in `c3` (Paid Out) or `c4` (Paid In), AND
- the description does not look like a fee row (`transaction fee`, `non-sterling`)

→ the current transaction is pushed and a new implicit transaction is started.
Its `type` is `''` (unknown — the code was not emitted by Azure DI); the column
position determines direction; the balance solver validates.
